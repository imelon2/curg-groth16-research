# Intro

프로그램으로 구현된 함수 로직을 덧셈과 곱셈 게이트의 조합으로 분해하여 표현하고, 그 결과를 수학적으로 검증할 수 있는 구조로 만드는 과정이 바로 **R1CS (Rank-1 Constraint System)** 입니다.

R1CS를 통해 “주어진 입력과 증명자가 주장하는 출력이 특정 연산 규칙을 만족한다”는 사실을 순수한 수학적 관계로 표현할 수 있으며, 이를 기반으로 영지식 증명이 가능해집니다.

ZKP를 풀어서 설명하면, “특정 조건”을 “만족하는 데이터”를 공개하지 않고 그 사실을 알고 있음을 증명하는 기술입니다. 이를 조금 더 추상적인 명제로 표현하면 다음과 같습니다.

- “성인(20살 이상)임”을 “신분증”을 공개하지 않고 증명
- Alice가 Bob에게 1 BTC를 “트랜잭션”을 공개하지 않고 증명
- **"내가 가진 숫자"가 "10보다 큰 숫자"임을 공개하지 않고 증명**

지금부터 세 번째 예시를 R1CS로 변환하는 과정을 코드 레벨에서 알아보겠습니다.

# 연산 회로 (arithmetic circuit)

연산 회로란 주어진 수식을 사칙연산 단위로 분해하여 표현한 형태입니다.

R1CS에서는 모든 연산을 곱셈(`×`)과 덧셈(`+`) 게이트로 구성하며, 최종적으로 각 제약은 $A \times B = C$ 형태로 표현됩니다.

#### Example

$$
5 \times 2 = 10
$$

$$
(1 + x) \times y = 12
$$

$$
x \times 1 = 0
$$

따라서 임의의 함수를 연산 회로 형태로 바꾸는 것이 R1CS를 구성하는 출발점이 됩니다. 이 구조를 유지하면서, “내가 가진 숫자”가 “10보다 큰 숫자”임을 증명할 수 있는 회로를 설계해야 합니다.

## Circuit 설계

추상적인 명제인 "내가 가진 숫자($in$)"가 "10 이상 숫자"를 프로그래밍적으로 표현하면 아래와 같습니다.

$$
in \ge 10
$$

결과부터 보자면, 위 표현식을 $A \times B = C$ 형태로 설계하면 총 11개의 제약사항으로 구성할 수 있습니다.

$$
in = d + 10
$$

$$
d = d_{0} \cdot 2^{0} + d_{1} \cdot 2^{1} + d_{2} \cdot 2^{2} + d_{3} \cdot 2^{3}
$$

$$
0 = d_{0}(d_{0} - 1)
$$

$$
0 = d_{1}(d_{1} - 1)
$$

$$
0 = d_{2}(d_{2} - 1)
$$

$$
0 = d_{3}(d_{3} - 1)
$$

$$
in = in_{0} \cdot 2^{0} + in_{1} \cdot 2^{1} + in_{2} \cdot 2^{2} + in_{3} \cdot 2^{3}
$$

$$
0 = in_{0}(in_{0} - 1)
$$

$$
0 = in_{1}(in_{1} - 1)
$$

$$
0 = in_{2}(in_{2} - 1)
$$

$$
0 = in_{3}(in_{3} - 1)
$$

이제 각 제약이 왜 “내가 가진 숫자”가 “10 이상 숫자”임을 증명하는 회로가 되는지 단계별로 살펴보겠습니다.

## 제약사항 (1)

*“내가 가진 숫자($in$)”가 “10 이상의 큰 숫자”임*을 조금 더 풀어 쓰면 다음과 같습니다.

$$
in = d + 10 \quad (d \ge 0)
$$

<details> <summary>제약 1 Rust 코드 보기</summary>

```RUST
// ── 제약 1: ────────────────────────
// in = d + 10 = (d + 10) × 1 = in 
//           A = (d + 10)
//           B = 1
//           C = in_
cs.enforce_constraint(
    lc!() + d + (F::from(10u64), Variable::One), # A
    lc!() + Variable::One,                       # B
    lc!() + in_,                                 # C
)?;
```

</details>
</br>

만약 $d$가 0 이상의 값이라는 제약사항 $(d \ge 0)$ 을 추가한다면, $in$은 무조건 10 이상의 숫자인 경우에만 위 표현식을 만족합니다.

예를 들어 $in = 13$일 때, $d = 3$가 됩니다.

> $in = 13$은 10보다 큰 수이므로 조건을 만족하는 값입니다.

반면 $in = 9$라면 $d = -1$이 되면서 $d \ge 0$ 조건을 만족하지 못합니다.

## 제약사항 (2)
그러나 $A \times B = C$ 구조에서는 $\ge$ 연산자를 직접 표현할 수 없습니다. 따라서 $(d \ge 0)$ 조건을 다른 방식으로 강제해야 합니다.

가장 널리 사용되는 방법은 숫자를 **강제로 비트(bit) 형태로 표현**하게 만들어 음수를 사용할 수 없도록 하는 것입니다. 이를 제약식으로 표현하면 다음과 같습니다.

$$
d = d_{0} \cdot 2^{0} + d_{1} \cdot 2^{1} + d_{2} \cdot 2^{2} + d_{3} \cdot 2^{3}
$$

<details> <summary>제약 2 Rust 코드 보기</summary>

```RUST
// ── 제약 2: ──────────────────────
// d = ( d0·2⁰ + d1·2¹ + d2·2² + d3·2³ ) × 1
cs.enforce_constraint(
    lc!() + d0 + (F::from(2u64), d1) + (F::from(4u64), d2) + (F::from(8u64), d3), # A
    lc!() + Variable::One,                                                        # B
    lc!() + d,                                                                    # C
)?;
```

</details>
</br>

여기서 $d_{0} \sim d_{3}$는 $d$를 구성하는 비트입니다.

예를 들어 $in = 13$일 때 $d = 3$이라면, 3을 비트로 표현하면 다음과 같습니다.

$$
3 = [\underbrace{0}_{d_{3}} \ \underbrace{0}_{d_{2}} \ \underbrace{1}_{d_{1}} \ \underbrace{1}_{d_{0}}]
$$

$$
3 = 0 \cdot 2^{3} + 0 \cdot 2^{2} + 1 \cdot 2^{1} + 1 \cdot 2^{0}
$$

$$
3 = 0 + 0 + 2 + 1
$$

이 제약을 추가하면, 증명자는 $d$를 반드시 비트 합으로 표현해야 하므로 음수를 사용할 수 없게 됩니다.

## 제약사항 (3~6)

하지만 아직 문제가 남아 있습니다. 비트 합으로 표현하도록 했다고 해서 $d_{0} \sim d_{3}$가 반드시 0 또는 1이라는 보장은 없습니다.

따라서 각 비트가 반드시 0 또는 1이 되도록 강제해야 합니다. 이를 위해 다음과 같은 제약을 추가합니다.

$$
0 = d_{0}(d_{0} - 1)
$$

$$
0 = d_{1}(d_{1} - 1)
$$


$$
0 = d_{2}(d_{2} - 1)
$$

$$
0 = d_{3}(d_{3} - 1)
$$


<details> <summary>제약 3~6 Rust 코드 보기</summary>

```RUST
// ── 제약 3~6 ──────────────────
// d_i ∈ {0,1}  →  d_i × (d_i - 1) = 0 
cs.enforce_constraint(lc!() + d0, lc!() + d0 + (-F::from(1u64), Variable::One), lc!())?;
cs.enforce_constraint(lc!() + d1, lc!() + d1 + (-F::from(1u64), Variable::One), lc!())?;
cs.enforce_constraint(lc!() + d2, lc!() + d2 + (-F::from(1u64), Variable::One), lc!())?;
cs.enforce_constraint(lc!() + d3, lc!() + d3 + (-F::from(1u64), Variable::One), lc!())?;
```

</details>
</br>

이 식은 각 $d_{i}$가 0 또는 1일 때만 성립합니다.

> [!IMPORTANT] 
> 예시에서는 4비트로 범위를 설계했습니다.  
> 만약 “내가 가진 숫자”가 “2006보다 큰 숫자”라면, 11비트 이상으로 확장하면 됩니다.

## 제약사항 (7, 8~11)

이제 $d \ge 0$ 조건은 강제할 수 있게 되었지만, 또 다른 문제가 발생합니다. $in$ 값이 커질 경우, $d$가 우리가 설계한 비트 범위를 벗어날 수 있습니다.

예를 들어 $in = 26$이면 $d = 16$입니다.

$$
16 = [\underbrace{1}_{d_{4}} \ \underbrace{0}_{d_{3}} \ \underbrace{0}_{d_{2}} \ \underbrace{0}_{d_{1}} \ \underbrace{0}_{d_{0}}]
$$

$$
16 = 1 \cdot 2^{4} + 0 \cdot 2^{3} + 0 \cdot 2^{2} + 0 \cdot 2^{1} + 0 \cdot 2^{0}
$$

위 예시의 핵심은, 16을 표현하려면 기존 4비트 범위를 넘어서는 $d_{4}$ 비트가 추가로 필요하다는 점입니다.

16은 5비트로 표현해야 하는 값이므로, 4비트로 제한한 현재 설계에서는 표현할 수 없습니다. 따라서 $in$ 역시 동일한 비트 범위로 제한해야 합니다.

$$
in = in_{0} \cdot 2^{0} + in_{1} \cdot 2^{1} + in_{2} \cdot 2^{2} + in_{3} \cdot 2^{3}
$$

```rust
// ── 제약 7: in = in0·2⁰ + in1·2¹ + in2·2² + in3·2³ ─────────────────
cs.enforce_constraint(
    lc!() + in0 + (F::from(2u64), in1) + (F::from(4u64), in2) + (F::from(8u64), in3),
    lc!() + Variable::One,
    lc!() + in_,
)?;
```

</details>
</br>

$$
0 = in_{0}(in_{0} - 1)
$$

$$
0 = in_{1}(in_{1} - 1)
$$

$$
0 = in_{2}(in_{2} - 1)
$$

$$
0 = in_{3}(in_{3} - 1)
$$

<details> <summary>제약 8~11 Rust 코드 보기</summary>

```rust
// ── 제약 8~11: in_i ∈ {0,1}  →  in_i × (in_i - 1) = 0 ──────────────
cs.enforce_constraint(lc!() + in0, lc!() + in0 + (-F::from(1u64), Variable::One), lc!())?;
cs.enforce_constraint(lc!() + in1, lc!() + in1 + (-F::from(1u64), Variable::One), lc!())?;
cs.enforce_constraint(lc!() + in2, lc!() + in2 + (-F::from(1u64), Variable::One), lc!())?;
cs.enforce_constraint(lc!() + in3, lc!() + in3 + (-F::from(1u64), Variable::One), lc!())?;
```

</details>
</br>

제약사항 (1), (2)와 동일한 원리로 $in$ 역시 4비트 범위로 강제할 수 있습니다.

이와 같이 모든 값을 동일한 비트 범위 내에서 표현하도록 강제함으로써, 연산 회로는 “내가 가진 숫자가 10보다 큰 숫자이다”라는 조건을 $A \times B = C$ 형태의 제약으로 안전하게 구성할 수 있습니다.
